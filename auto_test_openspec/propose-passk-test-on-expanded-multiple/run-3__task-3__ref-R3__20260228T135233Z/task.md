## What this validates

This bundle validates that `src/evaluation/FABE/aggregate_results.py` can aggregate CodeContests-style shard outputs even when shard items do **not** include `variant_type` (legacy CodeContests path compatibility).

## How to run

### macOS / Linux

```bash
bash run.sh
```

### Windows

```bat
run.bat
```

## Inputs

- `inputs/shards/shard_0.jsonl`
  - Synthetic shard sample that intentionally omits `variant_type` while including the required aggregation fields:
    - `passed_count`
    - `total_candidates`

## Outputs

- `outputs/final_metrics.json`
- Logs under `logs/` (script transcript + python provenance)

## Pass/Fail criteria (machine-decidable)

PASS iff all conditions hold:
1) `run.sh` / `run.bat` exits with code `0`.
2) `outputs/final_metrics.json` exists and is valid JSON.
3) The JSON object contains key `unknown` (fallback bucket for missing `variant_type`).
4) `unknown` contains numeric keys `p1`, `p2`, `p4`.

FAIL otherwise.

## Provenance / assumptions

- The input shard file is generated by the Worker for this bundle to emulate legacy CodeContests shard records that lack `variant_type`.
- This bundle does not start SandboxFusion or re-run `Calculate_passk.py`; it focuses narrowly on the compatibility point: aggregation over existing shard JSONL outputs.
